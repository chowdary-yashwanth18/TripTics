## ⏱️ 1-Minute Project Explanation (Elevator Pitch)

> *Read this out loud until it feels natural. Keep it under 60 seconds.*

"TripTics is a budget-friendly trip planning web app built with Python and Flask.
The idea is simple — a student or young traveler enters their total budget, trip duration,
number of people, and their interests, and the app recommends the top 3 Indian destinations
that fit their budget.

What makes it a data analytics project is the engine underneath:
the app loads a dataset of 125 Indian destinations with estimated hotel, food, transport,
and travel costs. It calculates the exact trip cost for every destination, scores each one
using a 100-point rule-based model, and returns the best matches sorted by score.

There's also a full analytics dashboard with 7 interactive charts and auto-generated insights
so the user can explore travel cost trends across states, trip types, and budget levels."

---

## ⏱️ 2-Minute Technical Explanation

### 1. Dataset Structure
- The dataset is a CSV file with **125 rows** and **12 columns**
- Each row represents one Indian travel destination
- Key cost columns: `Avg_Travel_Cost`, `Avg_Hotel_Per_Day`, `Avg_Food_Per_Day_Per_Person`, `Avg_Local_Travel_Per_Day`
- Additional metadata: `Type`, `Budget_Level`, `Best_Season`, `Crowd_Level`, `Family_Friendly`
- Loaded using **Pandas** with type enforcement in `utils/data_loader.py`

### 2. Cost Calculation Logic
The total estimated trip cost is calculated with this formula:

```
Total = Travel + (Hotel × days) + (Food × travelers × days) + (Local Transport × days)
```

- `Travel` is a one-time cost (paid once, not multiplied by days)
- `Hotel` is per night (multiplied by days)
- `Food` is per person per day (multiplied by both travelers and days)
- `Local Transport` is per day (multiplied by days)

This is implemented in `calculate_trip_cost()` inside `utils/recommender.py`.

### 3. Recommendation Scoring Logic
Each destination gets a **score out of 100** based on 4 factors:

| Factor | Points |
|---|---|
| Trip type matches user selection | 30 |
| Destination fits budget comfortably (75–100% usage) | 25 |
| Trip duration close to `Recommended_Days` | 25 |
| Budget level matches user style | 20 |

Destinations within budget are sorted by score (highest first).
If no destination fits the budget, the 3 closest over-budget alternatives are shown
along with plain-English suggestions like "increase budget by ₹X" or "reduce by 1 day."

### 4. Analytics Dashboard
Generated by `utils/analytics.py` using **Plotly**:
- 7 interactive charts (bar, grouped bar, donut, horizontal bar)
- 4 KPI summary cards (calculated live from dataset using Pandas `groupby`, `mean`, `min`)
- 5 key insights auto-generated from data aggregations

### 5. Flask App Flow
```
User visits / → index.html (landing page)
User submits /planner form → app.py validates input → recommender.py runs
→ results.html renders top 3 recommendations

User visits /dashboard → analytics.py generates 7 charts + KPIs + insights
→ dashboard.html renders all sections
```

The Flask app (`app.py`) acts as the orchestrator.
Business logic is cleanly separated into the `utils/` package.

## 🎓 Viva Q&A Preparation

### Q1: What is the objective of your project?
> "TripTics helps budget-conscious travelers find affordable Indian destinations
> based on their budget, group size, trip duration, and travel preferences.
> The system uses data analytics — cost modeling, scoring, and visualization — to
> generate smart recommendations without relying on any external APIs."

---

### Q2: Why did you choose CSV instead of a database?
> "For a semester project of this scope, a CSV is ideal. It is easy to read, modify,
> and understand. Pandas reads it directly without any database setup or SQL knowledge.
> The dataset is static (no live updates needed), so a database like SQLite would add
> complexity without real benefit. In a production version, we would migrate to a proper
> database and add admin tools to update cost data."

---

### Q3: How are recommendations generated?
> "The engine loops through all 125 destinations, calculates the exact trip cost for
> each using the formula: Travel + Hotel×days + Food×travelers×days + Local×days.
> Then it scores each destination out of 100 based on type match, budget comfort,
> duration relevance, and budget level match. Finally, destinations within budget
> are sorted by score and the top 3 are returned."

---

### Q4: What makes this a data analytics project?
> "Several core analytics concepts are applied:
> - **Data wrangling** with Pandas (loading, type coercion, normalization)
> - **Descriptive statistics** (mean, min, groupby aggregations for KPI cards)
> - **Rule-based scoring model** (instead of ML, we use transparent weighted scoring)
> - **Data visualization** (7 Plotly charts: bar, grouped bar, donut, horizontal bar)
> - **Data storytelling** (Key Insights section auto-derived from dataset trends)"

---

### Q5: How is the budget estimated?
> "The budget estimation uses a formula based on the dataset values. Travel cost is
> a one-time per-trip cost. Hotel cost is multiplied by the number of days. Food is
> multiplied by both days and number of travelers. Local transport is per day.
> These values are averages in the CSV — they are estimates, not live prices."

---

### Q6: What is the scoring system and why did you use it?
> "Instead of machine learning, we used a transparent 100-point rule-based scoring
> system. This has two advantages: it is easy to explain (students and faculty can
> verify the logic), and it is deterministic — the same inputs always give the same
> output. The 4 factors are weighted based on what matters most to a budget traveler:
> type preference (30 pts), budget fit (25 pts), days match (25 pts), style (20 pts)."

---

### Q7: What happens if no destination matches the budget?
> "The system shows the 3 closest over-budget alternatives. It also generates
> 3 actionable suggestions: how much extra budget is needed, how much could be
> saved by reducing the trip by 1 day, and a suggestion to widen the type/style filters."

---

### Q8: What are the limitations of the system?
> "The main limitations are:
> 1. Costs are static estimates — not live pricing
> 2. Only covers 7 Indian states (125 destinations)
> 3. No ML — rule-based scoring may miss nuanced preferences
> 4. No user accounts or trip saving
> 5. Starting city input is collected but not used in cost calculation yet"

---

### Q9: How can this project be improved in the future?
> "Key improvements would be:
> 1. Live cost API integration (MakeMyTrip, IRCTC)
> 2. User authentication and saved trip history
> 3. Machine learning recommendation using collaborative filtering
> 4. Expanding the dataset to all 28 states
> 5. PDF trip itinerary export
> 6. Travel time calculation from starting city using distance APIs"

---

### Q10: Why Flask instead of Django or another framework?
> "Flask is a lightweight micro-framework that is ideal for a project of this scope.
> It has minimal boilerplate, making the code easy to read and explain in a viva.
> Django would be overkill — it is better suited for large applications with ORM,
> admin panels, and complex authentication. Flask lets you see exactly what is
> happening at every step, which is important for a learning project."

---

### Q11: What is Plotly and why did you use it over Matplotlib?
> "Plotly generates interactive charts — users can hover over bars and slices to
> see exact values, zoom in, and pan. Matplotlib only generates static image files.
> For a web dashboard, interactive charts provide a far better user experience.
> Plotly charts are embedded as HTML divs directly in the Flask template using
> `pio.to_html(fig, full_html=False)` — no file saving needed."

---

### Q12: Explain the Flask app flow from form submission to results.
> "When the user submits the planner form, the POST request hits the `/planner`
> route in `app.py`. The form values (budget, days, travelers, trip type, style)
> are read and validated server-side. Then `recommend_destinations()` is called
> from `utils/recommender.py`. It calculates and scores all 125 destinations,
> returns the top 3 matches and any alternatives. These are passed to
> `results.html` via `render_template()` and rendered using Jinja2 template syntax."
